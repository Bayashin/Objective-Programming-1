はじめに

ファイル処理の基本から押さえよう

Javaで入出力処理、とくにファイル処理を行う場合、ファイルのデータすべてを変数へ一度に読み込ませるようなことは通常行いません。

理由は、単にファイルのサイズが非常に大きかった場合（たとえば2GByteとかそれ以上）、読み込んだタイミングでメモリが足りなくなってしまう可能性があるからです。 メモリが足りなくなってしまった際に出るエラーには、OutOfMemoryErrorというプログラムの実行が即座に止まってしまうエラーがあります。 これはプログラム自体にとって致命的な例外で、エラーが出た場合には回復の見込みがなく、プログラムが強制終了してしまうことは前回の授業でお話しました。

ではどうやってファイルを扱うのか、それはファイルのデータ自体を「少しずつ読む・書く」ことで解決できます。
この、「少しずつ読む・書く」ことはストリーム(stream) という考え方です。

ストリームとは

データの流れのことです。

例えて言うなら、ファイルなどの大量なデータが小川を少しずつ流れるようなものです。
プログラムは、この小川の流れからデータを読み取る、もしくは小川へデータを流し込む（書き出す）などして、少しずつ処理をします。

Javaでは、ファイル以外にも、標準入出力（System.outやSystem.in)などもストリームで処理を行います。 また、Javaに限らず他の言語でもストリームを扱える言語は数多く存在します。

Java SE 6 以前と 7 以降の入出力処理

ここで紹介するファイル処理はごく一部の機能です。 用途よっては異なる機能を使用しなければならない場合もあります。その点ご了承ください。

※Java8からのラムダで使用するようなStream APIとは違います。

Java8には、コレクションをストリームとして扱い、ラムダで処理する記法が追加されましたが、今回お話するストリームはそれとは違います。

Java SE 6以前ファイル読み込み処理

もっとも原初的な方法で、ファイルhoge.txtから1byteずつデータを取り出し、画面に出力するサンプルソースは以下のようになります。

//import java.io.*;
String filename = "hoge.txt";
FileReader fr = null;
try {
    //ファイルオープン
    fr = new FileReader(filename);
    int r;
    //fr.read()はファイルに含まれる次の1文字をint型で値を返します。
    //intで取得した値をchar型にキャストすることで文字になります。
    //EOFで-1を返します。(-1がきたらファイルの最後まで来たと言うこと)
    while((r = fr.read()) != -1) {
        //1文字ずつ画面に表示
        System.out.print((char)r);
    }
} catch (IOException e) {
    //入出力時例外
    //ファイルが見つからなかったなどの例外発生時に画面にメッセージを表示し終了
    System.out.println(e.getMessage());
    System.exit(1); //プログラムの異常終了
} finally {
    //ファイルクローズ
    try {
        if (fr != null) fr.close();
    } catch (IOException e) {
        //入出力時例外
        //ファイルを閉じる処理以前に、ファイル処理中で何らかの原因によりファイルが勝手に閉じられてしまった場合などに例外が発生します。
        System.out.println(e.getMessage());
        System.exit(1); //プログラムの異常終了
    } finally {
        //ファイルストリームの変数にnullを代入し、GCによるメモリの開放を促します。
        fr = null;
    }
}

C言語等の言語でも同じですが、ファイルを開いた後、何らかの処理を実行したら、ファイルは必ず閉じる 処理が必要です。

ファイル読み込み用のストリームFileReaderは使用し終わったら必ずclose()メソッドを呼び出さなければ、プログラムがファイルを開いた状態でロックしてしまい、他のプログラムで開くことができなくなってしまいます。

しかも、取り出せる文字は1文字ずつです。

ストリームを扱う場合は、このように確実な後始末をする必要があります。

これをもう少し発展させてみましょう。
以下のサンプルソースは、バッファストリーム (BufferedStream)を用いて、1行ずつ ファイルを読み込む処理です。

//import java.io.*;
String filename = "hoge.txt";
FileReader fr = null;
BufferedReader br = null;
try {
    //ファイルオープン
    fr = new FileReader(filename);
    //バッファを経由させ、以降はバッファから読み込み
    br = new BufferedReader(fr);
    String line;
    //br.readLine()は、EOFでnullを返す。
    while((line = br.readLine()) != null) {
        //読み込まれた1行分の文字列には改行は含まれない。
        System.out.println(line);
    }
} catch (IOException e) {
    //ファイルが見つからなかったなどの例外発生時に画面にメッセージを表示し終了
    System.out.println(e.getMessage());
    System.exit(1); //プログラムの異常終了
} finally {
    //ファイルクローズ
    try {
        if (fr != null) fr.close();
    } catch (IOException e) {
        //入出力時例外
        //ファイルを閉じる処理以前に、ファイル処理中で何らかの原因によりファイルが勝手に閉じられてしまった場合などに例外が発生します。
        System.out.println(e.getMessage());
        System.exit(1); //プログラムの異常終了
    } finally {
        fr = null;
    }
    //バッファクローズ
    try {
        if (br != null) br.close();
    } catch (IOException e) {
        //入出力時例外
        //ファイルを閉じる処理以前に、ファイル処理中で何らかの原因によりファイルが勝手に閉じられてしまった場合などに例外が発生します。
        System.out.println(e.getMessage());
        System.exit(1); //プログラムの異常終了
    } finally {
        br = null;
    }
}

複数のストリームを確実に開放する処理が増え、さらに面倒くさい雰囲気になりました。
ただ、テキストが1行ずつ読み込めるようになったことを考えると先ほどのソースコードよりは幾分かマシです。

(新)ファイル読み込み処理

古い書き方は、実際多くのプログラマを混乱させ、数多くのバグを発生させる悩みのタネでした。
それを改善するために、新しいバージョンのJavaが出るたびに改良されてきました。Java SE 7で新しくなった書き方を見てみましょう。

//import java.io.BufferedReader;
//import java.io.IOException;
//import java.io.UncheckedIOException;
//import java.nio.file.Files;
//import java.nio.file.Paths;
String filename = "hoge.txt";
//try-with-resources構文を用いてclose()を省略
try (BufferedReader br = Files.newBufferedReader(Paths.get(filename))) {
    String line;
    while ((line = br.readLine()) != null) { // 1行読み込みとNull比較を同時に行う
        System.out.println(line);
    }
} catch (IOException e) {
    // 本来はファイルが見つからない、ロックされてて開けないなどのファイル入力時例外として捕捉されることが想定されます。
    // UncheckedIOExceptionは、検査例外であるIOExceptionを非検査例外としてラッピングし、送出し直すための例外クラスです。
    throw new UncheckedIOException(e);
}

非常にシンプルになりました。
これなら間違いも少なくなりそうです。

この、リソースを指定して行う try 構文は、 try-with-resources構文 と呼ばれ、1つ以上のリソースを宣言するtry構文です。
BufferedReaderなどのストリームのリソースは、プログラムでの使用が終わったら閉じられなければいけないオブジェクトです。
try-with-resources構文は、try ( )内で宣言とインスタンス化をしたリソースについて、こちらで明示的に書かなくてもtry文の終わりで内部的に各リソースが確実に閉じられるようにします。finallyが要らなくなったということですね。

また、try ( )内には、複数のリソースを指定できます。その場合は、通常の行末と同じように;で区切りましょう。

Files.newBufferedReader(file.toPath())の部分については、FilesのstaticなメソッドnewBufferedReaderを用いてBufferedReaderを生成しています。FileReaderなどのファイル読み込み用ストリームを用意する必要がなくなりました。

さらにjava.util.ScannerでBufferedReaderを読み込むようにすると、while文の条件を簡略化できます。

// BufferedReaderのインスタンスは、Scannerの初期化にのみ使われるので変数定義は不要
try (Scanner sc = new Scanner(Files.newBufferedReader(Path.of(filename)))) {
    while (sc.hasNextLine()) {      // ここが簡略化される
        String line = sc.nextLine();
        System.out.println(line);
    }
} catch (IOException e) {
    throw new UncheckedIOException(e);
}

余談：try-with-resourcesでリソースとして使える条件

try-with-resourcesの構文内で、リソースとして指定できるオブジェクトは、java.io.Closeableインターフェイスを実装しているすべてのオブジェクトも含め、java.lang.AutoCloseableインタフェースを実装しているオブジェクトです。

Stream APIとラムダを用いた記法

java.nio.file.Filesには、テキストファイルの読み込み時、ストリーム処理を内包して利用者にそれを見せず、Java 8から実装されたStreamクラスを返す便利なメソッドが存在します。

//import java.io.IOException;
//import java.io.UncheckedIOException;
//import java.nio.file.Paths;
//import java.nio.file.Files;
try {
  Files.lines(Paths.get("hoge.txt")).forEach(l -> {
      System.out.println(l);
  });
} catch(IOException e) {
  throw new UncheckedIOException(e);
}

BufferedReaderを使用しないので、try-with-resources構文すら使用する必要もなくなりました。

ファイルパス・ファイル操作関連クラス

パスやファイルの操作を行う際に、便利なメソッドを持ったクラスがいくつかあります。

java.io.Fileクラス

Fileクラスのインスタンスには、指定されたパスにファイルが存在するかを確認するexists()メソッドや、読み込み可能か確認するcanRead()メソッドや、ファイル生成・消去などの制御を可能とするメソッドがあります。こういったメソッドを使用すると「ファイルが存在しない」などの例外をチェックすることや、空のファイルを作成するなどができるようになります。

参考：File (Java SE 17 & JDK 17 )

java.nio.fileパッケージ

Javaのnew I/Oパッケージです。

Javaにおけるファイル処理は、度重なる仕様変更によってどんどん便利な機能が追加されています。java.nio.fileパッケージは、Java SE 4以降に追加されたパッケージで、この内容はJavaのバージョンが新しくなるに連れてクラスやメソッドが追加されています。

上記のサンプルソースなどで用いられているFilesやPathsなどのクラスが含まれており、FilesやPathsなどはとくに新しいためnew I/O 2などと呼ばれます。
new I/O 2は、今まで煩雑だった書き方のファイル処理を簡潔に書けるようになるいくつかのファイル関連操作クラスが含まれています。

たとえば、Filesクラスや、Pathsクラスのstaticなメソッドを利用して、指定したパスのテキストファイルが読み込みできるかどうかを判断するには以下のようにできます。

if (!Files.isReadable(Paths.get("hoge.txt"))) {
    // 読み込み不可・・・
}

参考：java.nio.file (Java SE 17 & JDK 17)

(旧)ファイル書き込み処理

割愛します。読み込みと同様、非常に面倒です。

(新)ファイル書き込み処理

サンプルソースは以下のとおりです。

//import java.io.*;
//import java.nio.charset.Charset;
//import java.nio.file.Files;
//import java.nio.file.Paths;
//import java.nio.file.StandardOpenOption;
//import java.util.Arrays;

// 書き込みたいデータ
var lines = Arrays.asList("いろはにほへと", "ちりぬるを", "わかよたれそ", "つねならむ", "うゐのおくやま", "けふこえて", "あさきゆめみし", "ゑひもせすん");
try ( BufferedWriter bw = Files.newBufferedWriter(
        Paths.get("hige.txt"),                 // hige.txt
        Charset.defaultCharset(),                   // システムの標準文字コード(Macの場合UTF-8)で新規作成する
        StandardOpenOption.CREATE,                  // ファイルが見つからない場合は新規作成
        StandardOpenOption.WRITE,                   // 書き込みモード
        StandardOpenOption.TRUNCATE_EXISTING)       // ファイルがある場合は、データを消去する
) {
    for (String line : lines) {
        //一行書き込み
        bw.write(line);
        bw.newLine(); //改行を入れる
    }
} catch (IOException e) {
    throw new UncheckedIOException(e);
}

文字列の配列を、要素ごとに改行を入れながらテキストファイルに書き込みするサンプルです。 こちらも、読み込みと同様に、try-with-resources構文を用いてファイルクローズ処理を省略しています。

読み込みの際には省略してしまいましたが、newBufferedWriterメソッドを呼び出す時、文字コードをシステム標準の文字コードを指定するようにしました。 テキストファイルには、どうしても文字コードが異なることによる文字化けなど、嵌ってしまいやすい部分があります。 本来は、読み込み・書き込みどちらでも、文字コードを正しく指定することが望ましいのですが、都度書くのは手間がかかります。 文字化け等の不具合が起きる場合はこのように文字コードを指定できることを知っておきましょう。 文字コード自体の詳細については割愛しますが、テキストファイルには文字コードがつきものだ、と覚えておいてください。

また、java.nio.file.StandardOpenOptionには、他にもいくつかのオプションが存在しています。以下のドキュメントを見ておきましょう。

テキストファイルのアクセス方式

これまでのテキストファイルの入出力は、すべて特定の場所から順に読み込みを行う方式で行っています。
通常のテキストファイルに対する処理は、こういったシーケンシャルアクセス(Sequential Access) で対応できます。

しかし、「読み書きしたい場所とデータ」を都度指定するランダムアクセス(Random Access) をしたい場合もあります。

その様な場合には、ランダムアクセスを専用に行うAPIがあります。
詳しい説明は割愛しますが、java.io.RandomAccessFileクラスなどが使えます。

java.io.RandomAccessFileクラスのseek()メソッドでは、ファイルの先頭から何バイト目に移動するかどうかを決定してからreadByte()メソッドやwriteByte()メソッドで読み書きが可能です。

シーケンシャルアクセスとランダムアクセスのメリット・デメリット

それぞれ、先頭から順にアクセスする方法（シーケンシャルアクセス）と、指定した箇所を始めとしたアクセス方式（ランダムアクセス）では、それぞれメリット・デメリットがあります。その違いよ良く知っておきましょう。

シーケンシャルアクセス

メリット

  • 読み書きの速度が高速
  • 大きなファイルでも速度に違いが少ない

デメリット

  • 先頭から順にしか読み書きできない

ランダムアクセス

メリット

  • ファイルの読み書きしたい場所を指定することができる

デメリット

  • ファイルの変更によって、指定箇所のずれが生じる可能性あり
  • 読み書きの速度が遅い（指定箇所を探す処理などが必要なため）
  • 大きなファイルになると速度が顕著に遅くなる

とくに理由がなければシーケンシャルアクセスを使う

理由は、ファイルと言うプログラムでの管理外にある外部資源に対するアクセスを行うため、ランダムアクセスによるデメリットが大きくなります。
とくにjava.io.RandomAccessFileクラスのseek()メソッドを使った場合、先頭から何バイト目という方法でアクセスすることとなりますが、ファイルの変更によって変更箇所以降の読み込み・書き込み位置がずれていくことになります。ファイルを変更する都度、読み込みをし直す必要が出てきます。

ランダムアクセスを使用する場合には、操作したいファイルの特性をよく理解した上で扱うようにしましょう。 　

　

　

テキストファイルのさまざまな形式

単純なテキスト形式ファイルの他に、データを保存することに適した汎用フォーマットのファイル形式がいくつかあります。

代表的なものとしては、カンマ区切りでデータを羅列するCSV形式、Javaで使用されるプロパティファイル、タグによるデータへの意味付けを行いデータのネストが可能なXML形式、同様にネストが可能でWebでよく使用されるJSON形式などがあります。

それぞれの形式についてざっくりと解説致します。

CSV形式 (Comma-Separated Values)

データを以下のように,（カンマ）で区切るファイル形式です。

たろう,100,50

単純なテキスト処理で書き出しが可能でシンプルな形式のため、単純なデータを格納する際に用いられることが多いです。

データはすべて文字列形式で記録されます。
数値や日付などの値も文字列になって保存、取得されますので、数値の場合は変換処理が必要です。

一般的に、各データの区切り記号にはカンマが使用されますが、まれにスペースや、タブ（タブ区切りの場合はTab-Separated Values - TSVと呼びます）などを区切り記号とする場合もあります。
この区切り記号のことをデリミタ（delimiter）と呼びます。

CSVを扱うには、テキストファイルを1行ずつ読み込み、カンマごとに分割して処理を行います。
（この方法は、あくまでもJavaの標準APIを使ったもっとも簡単な方法で、融通が利きません。後述する方法も試してみましょう。）

String data = "たろう,じろう,さぶろう";  //データはCSVの1行を想定

String[] cols = data.split(",", -1);
for (String datum : cols) {
    System.out.println(datum);
}

Stringのインスタンスメソッドであるsplitを使うと、指定文字をデリミタとして分割しStringの配列を返します。

CSVファイルのフォーマット

CSV形式は、非常に古くから存在するファイル形式であり、数多くの方言が存在します。

たとえば、以下のような方言があります。

  1. 文字列は必ずダブルクォーテーションで囲う
  2. レコードの区切り文字をCR+LFとして、セル内改行はCRのみで表す
  3. ダブルクォーテーションで囲った文字列内であれば改行を入れて良い
  4. 文字コードはShift-JISのみ
  5. 1行目には必ずヘッダを設け、各列の名前を設定する
  6. #で始まる行はコメントとする

など、ちょっと考えただけでもどうやって実装すればよいか分からなくなるもの（とくに改行に関するもの※後述）があり、かつ全部の方言を仕様として制御しようと思っても矛盾が生じてしまうため不可能と言って良いです。
（金額項目に3桁区切りで,が入ってくるデータとか送られてくると軽く発狂します。）

統一の仕様については、RFC 4180で定められており、この仕様に準拠しようとすると、以下のCSVデータも正しく読まないといけません。

# これはコメント
"aaa","bbb
#コメントじゃない","ccc"

このデータは、以下の様なデータとして区切られます。

1 2 3
aaa bbb
#コメントじゃない		 ccc

先ほど紹介した1行ずつ読み込む、という処理方法で、この仕様は満たせるでしょうか？（当然ながら無理なのはわかると思います。）

では、RFC4180に準拠したCSV読み込みの方法はどうすれば良いのかというと、外部ライブラリに頼りましょう。という結論になります。

Javaプロパティファイル

CSV形式では、それぞれのデータの意味が分かりずらいです。 データにきちんと意味づけを行いたい場合はデータをキーと値のペアとして保存できる形式*.propertiesを使います。

javaのプロパティファイルに関するルールは以下のとおりです。

  1. データはすべて文字列として扱われ、テキストファイルとして保存される
  2. 各行には「キー」と「値」をペアとして、「ペアデリミタ」で区切り記述する
  3. ペアデリミタには、=（イコール）、:（コロン）、 （スペース）のどれかの文字を使用する
  4. #（シャープ）、!（感嘆符）で始まる行は、コメントとして無視される
  5. ファイルの拡張子は*.propertiesを使用するのが一般的だがテキストファイルであれば拡張子は関係ない

プロパティファイルの読み書きには、java.util.Propertiesクラスを使用します。

単純な単一データの保持にはテキストやCSVよりこちらを使ったほうが便利です。

Javaプロパティファイルは、以下のようにデータの名前とデータを「=」でつなぎます。

# プロパティファイル名は、「hoge.properties」です。
# 拡張子は.propertiesじゃなくてもOK。
# コメントは＃記号で
name=hoge
mail=hoge@aitech.ac.jp

プロパティファイルの読み込みは、以下のソースコードを参考にしてください。

//import java.io.FileInputStream;
//import java.io.IOException;
//import java.util.Properties;
String configFile = "hoge.properties";
Properties prop = new Properties();
try {
  // プロパティファイルの読込
  prop.load(new FileInputStream(configFile));
} catch (IOException e) {
  throw new UncheckedIOException(e);
}
if (prop.isEmpty()) {
  System.out.println("プロパティファイルの読み込みに失敗しました。");
}
String name = prop.getProperty("name");
String mail = prop.getProperty("mail");
System.out.println("name: " + name);
System.out.println("mail: " + mail);

XML形式 (eXtensible Markup Language)

XML形式は、データに意味付けを行うと同時にデータのネストを可能とするテキスト形式です。

たとえば、名前とHPと攻撃力と武器（名前と攻撃力）を持つ勇者のデータをXML形式で表すと以下のようになります。

<hero>
    <name>たろう</name>
    <hp>100</hp>
    <atk>50</atk>
    <weapon>
        <name>きのぼう</name>
        <rate>1.3</rate>
    </weapon>
</hero>

親子構造を保持するので、データへのアクセス方法は、若干特殊です。 上記の例で、hpを取り出す方法をまとめると、以下のようになります。

  1. ファイル全体を取得
  2. 一番外側のheroタグを探す
  3. その中のhpタグを取得
  4. hpタグの中身を文字列で取り出す

XML形式の取り扱いは、java.xml.parsersパッケージなどで構成されるAPIを使用します。

サンプルソースについては割愛します。

JSON形式 (JavaScript Object Nortation)

XML形式と似たようなデータ・フォーマットで、データのネストが可能です。
名前から見てもわかるように、もともとはJavaScriptのために用意されたデータ形式で、JavaScriptではこのデータ形式を自動的にインスタンス化してオブジェクトとして利用できるメリットがあります。
また、XMLに比べ軽量で、Webシステムなどでよく利用されるデータ・フォーマットです。

上記XMLサンプルと同じデータをJSONで表現すると以下のようになります。

{"hero": {"name": "たろう","hp":100,"atk":50, "weapon": {"name":"きのぼう","rate":1.3} } }

特徴としては、数値を数値として保持できるところと、データ量がXMLに比べ軽量であるところです。

ただ、Javaの強力な型システムとはかなり相性が悪く、JSONの柔軟な項目の表現をそのまま扱うことは難しいです。 サンプルソースについては割愛しますが、以下のライブラリを検討すると良いと思います。

  • google/gson: A Java serialization/deserialization library to convert Java Objects into JSON and back
  • FasterXML/jackson: Main Portal page for the Jackson project

バイナリファイルの読み書き

ファイルには、テキスト形式で書き込まれたファイルの他に、2進数の01で表現されるバイナリファイルがあります。

テキストファイルもバイナリファイルの内の1つではありますが、扱うデータが文字かバイナリかで処理方法が大きく異なります。

バイナリファイルを扱うストリーム

テキストファイルを扱うストリームとして、代表的なものにFileReaderやFileWriterがあります。

それに対して、バイナリファイルを扱うストリームは、代表的なものにFileInputStream、FileOutputStreamがあります。

FileInputStream,FileOutputStream

コンストラクタやメソッドは、FileReaderやFileWriterとよく似ていますが、取り扱うデータが文字からバイト配列に変わっています。

また、write()メソッドや、read()メソッドの読み書きの単位は、1バイト（8bit）ずつとなっています。

テキストデータをバイナリで操作する場合の注意点

テキストデータをバイナリで操作する場合には、文字コード に気をつける必要があります。

人間から見たら同じ あ という文字であったとしても、UTF-8で保存した場合は1110 0011 1000 0001 (0xE381) と、Shift-JISで保存した場合は1000 0010 1010 0000 (0x82A0) で値が異なります。
文字コードが異なれば、バイナリデータも異なってきます。とくに日本語のデータを扱う場合には注意が必要ですので覚えておきましょう。

バイナリデータの確認方法

バイナリエディタと呼ばれるバイナリデータの閲覧（または編集）が可能なツールを使います。

Macの場合は、App Storeで配布されている「Hex Field」がオススメです。

  • 「Hex Fiend」をMac App Storeで

または、VS Codeなどのテキストエディタの拡張機能として、バイナリエディタが配布されていたりします。 自分の環境に合わせてバイナリエディタを探してみるとよいでしょう。

テキストファイルの文字コードを指定して読み込み

文字コードを指定する場合、Files.newBufferedReaderにて第2引数で指定します。

以下はShift-JIS(ms932)の文字コードのファイルを読み込みます。

//import java.io.IOException;
//import java.io.UncheckedIOException;
//import java.nio.file.Files;
//import java.nio.file.Path;
//import java.util.Scanner;
//import java.nio.charset.Charset;

try (Scanner sc = new Scanner(Files.newBufferedReader(Path.of("path/to/file.txt"), Charset.forName("ms932")))) {
    while (sc.hasNextLine()) {
        System.out.println(sc.nextLine());
    }
} catch (IOException e) {
    throw new UncheckedIOException(e);
}

古い日本語テキストファイルには、Shift-JIS(ms932)という文字コードが使用されていることがあり、これを使うことで文字化けを回避できます。

また、Charset.defaultCharset()を使用するとシステム標準文字コードを指定でき、java.nio.charset.StandardCharsetsには、標準的に使用される文字コード数種類の定数が用意されています。

ここで紹介している読み込みメソッドには、このコードと同様に文字コードを指定して開くことができるものもあります。必要な場合は各自調べて使いましょう。

Shift-JISの文字コードを持ったファイル処理

Javaにおいて、Shift-JISとして使用できる文字コードには、Shift-JIS,cp932,ms932の3種類があり、それぞれ歴史的経緯によりShift-JIS → cp932 → ms932の順に新しくなっております。(cp932は正式名称ですが、ms932はcp932のJavaでの実装上の名称であり正式名称ではありません。)

また厄介なことに、JavaでShift-JISを指定してファイルを読み取りすると、MalformedInputExceptionによる文字コードミスマッチの例外は発生しない状況が存在します。
その1つが、BufferedReaderを使った際にShift-JISでは認識できないがms932では認識できる文字をShift-JISとして指定すると、その文字でEOFが発生してしまい、ファイルが最後まで読み込まれないことがあるという点です。

現在のJavaでは、基本的にShift-JISの文字コードのファイルはすべてms932で読み込めば問題なく読み取れます。

IDEを使った場合のデフォルトファイル参照先

たとえば、以下のソースコードでファイルを開こうとした場合、どのディレクトリのhoge.txtが読みこまれるか、という話です。

//import java.io.*;
//import java.nio.file.Files;
//import java.nio.file.Path;
String filename = "hoge.txt";
try (Scanner sc = new Scanner(Files.newBufferedReader(Path.of(filename)))) {
    while (sc.hasNextLine()) {
        String line = sc.nextLine();
        System.out.println(line);
    }
} catch (IOException e) {
    throw new UncheckedIOException(e);
}

この場合は、プロジェクトフォルダ直下のファイルが読みこまれます。

  • ［プロジェクト名ディレクトリ］
    • ［.idea］<- IntelliJ IDEAのプロジェクト設定ファイル等IDE用のファイルが格納されます
    • ［out］<- コンパイル済みの中間コードやjarファイルなどの成果物が入ります
    • ［src］<- プロジェクトのパッケージ、ソースコードが入ってます
    • ［プロジェクト名］.iml <- プロジェクトのビルドに関する設定ファイル
    • hoge.txt <- これが読みこまれます。
